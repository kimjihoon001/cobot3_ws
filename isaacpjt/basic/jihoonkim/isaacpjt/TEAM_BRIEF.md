**★기본 제공 `RidgebackUr`(UR5)로는 안 된다 — 0.4m 부족★**

통로 중앙에 서면 과실까지 **수평 0.66m** 를 써야 한다. 그러면 UR5 의 도달 0.85m 중
남는 수직이 ±0.54m 뿐이고, 어깨가 0.46m 이므로 **최대 1.0m 까지밖에 안 올라간다.**
과실은 **1.4m 까지** 있다. Franka(0.85m)도 같은 이유로 0.23m 부족.

| 팔 | 도달 | 수평 0.66m 쓰고 남는 수직 | 판정 |
|---|---|---|---|
| UR5e | 0.85m | −0.08 ~ **1.00m** | 0.40m 부족 |
| Franka | 0.85m | 0.09 ~ **1.17m** | 0.23m 부족 |
| **UR10e** | **1.30m** | −0.64 ~ **1.60m** | **OK** |

→ **Ridgeback + UR10e** 로 조립해야 한다. 둘 다 Isaac 에 있으니 팔만 갈아끼우면 된다.
→ 대안: 팔을 기둥 위에 올려 어깨를 높이는 것(예전 M0609@0.95m 안이 그거였다).

**이 결론이 좋은 이유:** "왜 UR10e냐" 에 답이 있다. 스펙이 좋아서가 아니라
**조간 1.5m(제주농기원) + 과실 높이 0.5~1.4m(인체치수)** 라는 근거 있는 값 두 개에서
유도된 결과다. 부품을 먼저 고르고 되는지 본 게 아니다.

**⚠ 전부 문서 기준이다. 실물 확인 필요:**
`isaac_python spikes/03_asset_check.py` (5분) — 에셋 존재·치수·통로 통과를 자동 확인.
위 도달 검산은 **구 근사**라 실제 도달영역의 모양과 자세 제약을 못 본다.

## 3. 의존 관계 — 병렬 가능/불가

```
  [에셋: 트레이/AMR/창고랙]  ─┐
                              ├─→ [씬 재구성: 6섹터+창고] ─→ [로봇 배치]
  [엔드이펙터 설계]          ─┘                                  │
                                                                 ├─→ 5★ Pick&Place
  [데이터셋 재생성] ─→ [YOLO 학습] ─→ 3★ Detection ─→ [pick point 3D] ─┘
                                                                 │
                                              6·7·8 트레이/배차 ──┤
                                                                 └─→ 9~11★ AMR 운반·하역
                                                                        │
                                                                        └─→ 통합
```

**병목 두 개:**
1. **에셋이 씬을 막고, 씬이 로봇을 막고, 로봇이 5·9~11번을 막는다.**
   → 에셋 3종이 Day1~2 에 안 나오면 그 뒤가 전부 밀린다. v3 11.1 배정이 이걸 노린 것.
2. **Detection 이 5번의 입력이다.** YOLO 없이 Pick&Place 를 개발하려면
   **정답 좌표(ground truth)로 임시 대체**해야 병렬이 된다.
   → 이걸 안 하면 Vision 담당이 끝날 때까지 MM 담당이 놀게 된다. **인터페이스만 먼저 합의하면 병렬 가능.**

**병렬 가능한 것 (서로 안 막음):**
- Vision 데이터셋·학습  ↔  에셋 모델링  ↔  트레이/배차/로거(개인 PC, Isaac 불필요)
- 브리지 검증은 로봇 없이도 가능 (스텁 로봇 있음)

## 4. 오늘 발견 — 범위에 영향

### 4.1 커터를 하드웨어에 추가해야 함 (v3 6.2 수정)

v3 하드웨어에 **그리퍼만 있고 커터가 없다** = 당겨서 뜯는다는 뜻이 된다.

> 근거 [W2024] Agronomy 2024, 14(10):2274 —
> 인장(당기기) 수확은 **손상 확률 최대**. 전단 성공률 **100%** vs 굽힘 42.83% → **shear 권장**.
> 절단 부위는 **distal pedicel** (전단력 33.241 N). proximal 은 **85.32% 더 힘듦** (Table 4).

→ 엔드이펙터 = **2-finger 그리퍼 + 커터 일체형**. 커터는 파지점에서 **위로 약 4~5cm**
  (과실 반지름 34mm + 꽃자루). 한 자세에서 잡고 자른다.

**vacuum 은 비추천:** Isaac 에 vacuum 이 없어서 고정 조인트로 흉내내야 하는데, 그러면
흡착력·마찰이 결과를 안 바꾸게 되고 "그 값 근거가 뭐냐"에 답이 없어진다.

**시뮬에서 커터는 물리가 필요 없다.** 칼날이 실제로 절삭하는 건 범위 밖.
위치만 맞으면 조인트를 끊는다. **잡는 건 진짜 마찰**이어야 한다.

### 4.2 트레이 사전 적재(50%)는 빼는 게 이득

v3 8장이 **스스로 약점이라고 인정**한 항목이다:

> "트레이 사전 적재가 처리량 검증을 **과소평가할 위험** — 트레이를 처음부터 다
> 채우는 전체 사이클을 보여주지 않음"

빼도 **개발 시간은 같다**(파라미터 하나). 대신 전체 사이클이 진짜 검증이 된다.
일정 리스크 때문에 넣은 단축인데, 비용이 거의 0인 곳에서 퀄리티를 깎고 있다.

**씬 규모(과실 401개)는 유지한다.** 사전 적재를 빼면 pick 사이클이 늘어나므로
과실이 많은 게 오히려 맞고, 정량 검증의 표본도 커진다. 401개가 물리적으로
도는지는 `spikes/02` 가 답한다 — 추측으로 미리 줄이지 않는다.
(되면 아무도 안 해본 규모라 그 자체가 발표 재료다. 안 되면 그때 하이브리드로.)

### 4.3 진짜 리스크에 아무도 손 안 댐

v3 10장이 꼽은 것:
- **포크 삽입 정렬 오차** — 문서가 "**새로운 리스크**"로 명시. AprilTag 보정을 필수로 승격.
- **MM↔AMR 핸드오프 동기화**

둘 다 크리티컬 포인트 9~11번에 직결인데 **코드도 검증도 0**이다.
반면 오늘 내가 본 건 마찰 파지·꽃자루였다 — 배점상 우선순위가 어긋나 있었다.

## 5. 미검증 — 공수 추정이 틀릴 수 있는 지점

**Isaac 에서 확인된 게 0개다.** 아래는 "될 것 같다"이지 "된다"가 아니다.
내일 GPU 앞에서 30분씩이면 답이 나오고, **그 답이 일정의 근거가 된다.**

| 확인할 것 | 왜 급한가 | 방법 |
|---|---|---|
| **Isaac 에 AMR/지게차/그리퍼 에셋이 있나** | 없으면 전부 직접 모델링 = 며칠. **최대 변수** | `spikes/03_asset_check.py` 5분 |
| **좁은 이랑에서 팔이 닿나** | 안 닿으면 온실 치수나 로봇 구성이 바뀜 | 로봇 배치 후 확인 |
| **ROS2 브리지가 붙나** | 안 붙으면 다중PC(3대) 구조가 무너짐 | `main.py` + `fsm_node` |
| 마찰 파지가 되나 | 안 되면 파지 방식 재검토 | `spikes/01` |
| RMPflow 제공 로봇 목록 | Pick&Place 개발 속도를 좌우 | 확인 |

**내일 아침 순서 제안:**
1. **`spikes/03_asset_check.py`** (5분) — 에셋 유무·치수·통로 통과. **회의 전에 돌릴 것.**
   ForkliftB 가 있으면 AMR 모델링 배정이 사라지므로 역할 배분 자체가 달라진다.
2. **브리지** (30분) — `main.py` + `fsm_node`. 다중PC 구조가 성립하나.
3. **`spikes/01`** (30분) — 필요 파지력 → 그리퍼 선정 근거.
4. 나머지는 역할 정해지고 각자.
