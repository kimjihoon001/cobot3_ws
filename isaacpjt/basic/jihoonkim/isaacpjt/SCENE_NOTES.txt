1. 근거가 확보된 값  ★발표에서 자신있게 쓸 것★
================================================================

[재식거리] 조간 1.5m / 주간 0.3m         -> config/settings.py PlantConfig
  근거: 제주특별자치도 농업기술원 「과채류(토마토)」 정식거리
        "배지경 양액재배시는 150×25~30cm 로 1열로 심는 것이
         통풍채광이나 작업이 편리하다"
  왜 이 값인가: 스마트팜 = 배지경 양액재배(코이어/암면 + 양액).
        토경 보통재배(90~100 × 40~50cm)와 값이 다르므로 헷갈리지 말 것.
  교차검증: TAMU / UMass protected culture high-wire 조간 1.5~1.8m
        재식밀도로 환산 시 1.5 × 0.3 = 0.45 m2/주
        UMass 권장 3.5~6 sq ft/주 = 0.33~0.56 m2/주 -> 범위 안에 들어감
  => 국내 자료와 미국 자료가 독립적으로 일치.

[그루 수] 4줄 × 33그루 = 132그루, 과실 401개
  ★설정값이 아니라 유도값★
  온실(8×12m) - 마진(1.2m) = 재배구역 5.6 × 9.6m
  5.6 / 1.5 -> 4줄,  9.6 / 0.3 -> 33그루
  => "왜 4줄이냐"에 대한 답: 우리가 정한 게 아니라 재식거리에서 나옴.
     정할 수 있는 여지 자체를 코드에서 없앰 (scene/tomato_plants.py _positions).

[수확 구간 높이] 0.5 ~ 1.4m              -> PlantConfig.fruit_height_range
  근거: 스마트팜 토마토는 하이와이어 + lower&lean(낮추기) 방식.
        "The ideal space for the harvest zone is a level even with the
         knee-to-shoulder range of the person who will be doing the harvesting"
        "plants are continuously lowered, the fruit remains at a convenient
         height for picking"
  핵심 논리: 과실 높이는 식물이 자란 높이가 아니라 '사람 인간공학'이 정한다.
        줄기를 계속 낮춰 수확 화방을 항상 무릎~어깨에 유지하기 때문.
  한국인 환산: 제8차 한국인 인체치수조사 평균 신장 남 172.5 / 여 159.6cm
        -> 무릎~어깨 약 0.45~1.45m
  ★발표 논거★ "로봇 편하라고 높이를 정한 게 아니라, 실제 온실이 사람 기준으로
        지어져 있어서 로봇이 거기에 맞춰야 한다" (실제 농업로봇의 제약 그대로)
  약점: 신장 비례 추정임. sizekorea.kr 에서 무릎높이/어깨높이 실측값을
        직접 뽑으면 한 단계 강해짐. (TODO)

[익음/불량 클래스] green / half_ripe / fully_ripe / old
  근거: isaac/tomatest/CHECKLIST.txt 참고 (Laboro=익음, VegNet=불량)
  요약: 익음 3클래스 = Laboro Tomato (USDA 색상등급 기반, 빨강 % 정량 정의)
        불량 old      = VegNet, Data in Brief 45:108657 (2022)
                        DOI 10.1016/j.dib.2022.108657, CC BY
                        클래스명도 VegNet 용어 "Old" 그대로 채택

[필요 리치] 통로 중앙 기준 0.95m -> 이동 베이스가 이랑 0.5m 접근 시 M0609 의 71%
  유도 과정:
    조간 1.5m / 2 + 과실 오프셋 0.09 = 통로 중앙에서 0.84m
    수확 구간 0.5~1.4m 의 중앙 = 0.95m 에 팔 밑동을 두면 리치 최소화
      (위/아래 과실까지 거리가 같아짐)
    베이스 높이별 필요 리치: 바닥 1.63m / 0.5m -> 1.23m / 0.95m -> 0.95m
    ★바닥 설치는 M0609(900mm)로 불가. 이동 베이스 위 0.95m 가 필수★
  이동 베이스가 이랑에 가까울수록 여유가 생김:
    이랑에서 0.75m(통로 중앙) -> 리치의 91% (빠듯)
    이랑에서 0.50m            -> 리치의 71% (여유)  <- 채택
  => 통로 1.5m 이므로 한 번에 한 줄 담당, 반대편은 돌아 나올 때.
     (실제 수확 로봇 방식)

[수확 방식] 당기기 X -> 꽃자루(distal pedicel) 절단 O
  근거: Tomato Pedicel Physical Characterization for Fruit-Pedicel Separation
        Tomato Harvesting Robot, Agronomy 2024, 14(10):2274
        DOI 10.3390/agronomy14102274 (MDPI 오픈액세스, CC BY)
        - proximal 꽃자루 전단력이 distal 보다 85.32% 큼
        - 전단 에너지는 121.37% 큼
        - "로봇이 수확 지점을 고를 때 distal pedicel 을 자르는 게 효율적"
  => CHECKLIST 의 "breakable joint 로 붙이고 그리퍼로 당겨 따기" 는 실제 방식이
     아님. 당기면 줄기 전체가 크게 변형되며 실패/충돌.
  부수효과: 자를 거면 그리퍼는 과실을 받치기만 하면 되므로 정밀 파지력 불필요.
           => "RG2 로 미세한 거 못 한다" 는 걱정이 사라짐.

3. 설계 판단 (왜 이렇게 했는지)
================================================================

[과실 = kinematic RigidBody]           scene/physics.py, tomato_plants.py
  과실 401개를 전부 dynamic 으로 두면 물리 솔버가 매 스텝 다 푼다.
  kinematic 이면 비용이 거의 0. 수확 순간에만 kinematic 을 꺼서 dynamic 전환.
  => CHECKLIST 의 "매달린 건 static, 수확되는 것만 dynamic (성능)" 을 구현.

[breakForce 안 씀 -> 코드로 joint 절단]  greenhouse_task.detach_fruit()
  breakForce(물리로 끊기)는 매번 결과가 미세하게 달라져 비결정적.
  평가지표: "재현성: Play/Stop 반복 실행 시 매번 동일한 초기 상태에서
             동일한 동작 결과가 재현된다" (10점 항목)
  => 코드로 kinematic 을 끄면 결정적. breakForce 는 '실패 모드' 용으로만 남김.
     (API 는 검증 완료: physics:breakForce / physics:breakTorque 동작함)

[콜라이더 = convexHull, 꼭지는 제외]
  트라이앵글 메시 충돌 금지(성능). 토마토는 볼록해서 convexHull 로 충분.
  꼭지(calyx)는 장식이라 콜라이더를 안 붙임 = 비용 절감.

[BaseTask 구조]                        scene/greenhouse_task.py
  SceneBuilder 를 대체함. 이유:
    - World 가 Play/Stop 시 post_reset() 을 불러줘 초기화가 한 곳에 모임
    - get_observations() 가 관측 인터페이스 표준이 됨
    - 강의 예제(M0609 pick&place)와 같은 구조 -> 로봇 Task 와 합치기 쉬움

[GroundTruthDetector 로 YOLO 를 임계경로에서 뺌]
  시뮬 안에는 정답이 이미 있다(우리가 만들었으니).
  get_observations()["fruits"] 가 {path, class_name, position} 을 준다.
  => YOLO 학습이 늦어져도 Detect->Pick->Place 플로우를 완성할 수 있음.
     나중에 같은 형식을 반환하는 YoloDetector 로 갈아끼우면 FSM 은 안 바뀜.
  ※ 평가지표상 AI 비전은 10점, 다중로봇/통합은 60점. 배점상으로도 맞는 판단.
